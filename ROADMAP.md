# Roadmap — Interview Helper

Тренажёр для подготовки к собеседованиям. Старт: трек «Аналитика», грейд Junior.
Долгосрочное видение: формат в духе Duolingo (привычка, streak, геймификация),
но для технических собеседований.

> **Принцип этого roadmap:** сначала проверяем, что продукт нужен людям,
> и только потом вкладываемся в инфраструктуру. Ориентир нагрузки на этапе
> проверки — десятки пользователей. Это сознательно удерживает от
> преждевременного усложнения (YAGNI).

> **Где мы сейчас:** рабочий MVP с локальным хранением. Структура контента
> уже доросла до `version 2` (tracks → grades → topics → questions),
> вопросы поддерживают множественный выбор (`correctIndexes`). Главные
> узкие места для проверки гипотезы — **мало вопросов** и **сырой UX**.
> Раздача первым людям — веб-сборка по ссылке, затем Google Play.

## Фаза 1 — Довести MVP до состояния «можно дать людям» ← мы здесь

**Цель фазы:** 5–10 реальных людей проходят тренажёр и говорят, полезно или нет.
**Контент и UX здесь важнее новых фич.**

### Контент (узкое место №1)
- [ ] Наполнить банк: 30–50 вопросов на каждую тему трека «Аналитика / Junior»
- [ ] Добиться, чтобы каждая тема ощущалась «полной», а не демо-набором
- [ ] Вычитать пояснения (`explanation`) — это главная ценность для готовящегося

### UX-полировка (узкое место №2)
- [x] Пустые состояния (нет прогресса, тема пройдена)
- [ ] Пустые состояния (нет интернета на web)
- [ ] Анимации переходов между вопросами и на экран результата
- [x] Обработка краёв: выход из сессии на середине, возврат
- [ ] Обработка краёв: быстрые тапы
- [x] Визуально показать слабые темы (данные `weakestTopics` уже есть в коде)
- [ ] Иконка приложения и финальное название

### Доставка первым пользователям
- [x] Собрать web-версию (`flutter build web`) и выложить на GitHub Pages
- [ ] Проверить работу на мобильном браузере (большинство откроют с телефона)
- [ ] Короткая ссылка + 2–3 строки «что это и как помочь фидбэком»

### Сбор обратной связи
- [ ] Способ оставить фидбэк прямо из приложения (ссылка на форму)
- [ ] 4–5 вопросов в форме: понятно ли, полезно ли, чего не хватило, сложность
- [ ] Поговорить лично хотя бы с 3 пользователями (живой фидбэк > формы)

**Критерий перехода к Фазе 2:** есть подтверждение, что формат полезен —
люди проходят больше одной сессии и говорят «да, помогло бы на собесе».

## Фаза 2 — Удержание без бэкенда (превращаем заходы в привычку)

**Цель фазы:** пользователь возвращается, не дожидаясь сервера.
Всё это реализуемо на локальном хранении — сервер пока не нужен.

- [ ] Дневная цель (например, N вопросов в день) и её визуализация
- [ ] Достижения / бейджи за streak и за закрытые темы
- [ ] Push-уведомления («не теряй streak», напоминание о цели) — локальные
- [ ] Экран статистики: графики по темам, динамика точности, слабые места
- [ ] Расширить контент по результатам Фазы 1 (темы, которые просили люди)

**Критерий перехода к Фазе 3:** есть возвраты (часть людей заходит повторно),
и контента/функций уже не хватает в рамках «всё на устройстве».

## Фаза 3 — Бэкенд и подключение Claude

**Цель фазы:** обновлять контент без релиза + оценивать открытые ответы.
Здесь впервые задействуется собственный сервер — и только когда Фазы 1–2
показали, что продукт нужен.

- [ ] Эндпоинт, отдающий банк вопросов (`GET /questions`)
- [ ] Переключить загрузку вопросов с ассета на HTTP (`http` уже подключён)
- [ ] Кэширование вопросов на устройстве (офлайн + скорость)
- [ ] Прокси-эндпоинт для Claude (`POST /evaluate`) — ключ только на сервере
- [ ] Открытые вопросы: пользователь пишет ответ → сервер → Claude → оценка
- [ ] Базовая защита эндпоинтов (rate limiting, простая авторизация)

**Важно по безопасности:** API-ключ Anthropic никогда не попадает в приложение.
Схема всегда: приложение → сервер → Claude API → сервер → приложение.

## Технические заметки

- **Стек:** Flutter (Dart). Один код под iOS, Android и web.
- **Хранение (сейчас):** локально через `shared_preferences`.
- **Контент:** `assets/data/questions.json`, формат `version 2`.
- **Сервер:** появляется в Фазе 3, не раньше.
- **Масштаб:** проектируем под десятки—сотни пользователей. Инфраструктуру
  «на вырост» не строим, пока нет реальной нагрузки.

## Принципы принятия решений

1. Не строим то, что ещё не нужно (YAGNI).
2. Контент и проверка гипотезы важнее технического совершенства.
3. Каждая фаза имеет смысл только если предыдущая дала результат.
4. Простое решение, которое работает сегодня, лучше идеального через полгода.
